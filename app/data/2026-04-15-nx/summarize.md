En este video, aprendí cómo usar NX para gestionar un monorepositorio y mejorar la arquitectura de proyectos, especialmente en Angular. Entendí que NX cobra mucho sentido cuando tengo múltiples aplicaciones, como un website público y un panel administrativo, que aunque sean proyectos separados, suelen compartir servicios, tipos y lógica. En lugar de duplicar código o mezclar todo en una sola app enorme, NX me permite mantener aplicaciones independientes dentro de un mismo repositorio, compartiendo librerías de forma ordenada. También vi que esto no solo aplica cuando tengo varias apps: incluso si solo tengo una aplicación Angular que crece con el tiempo, NX me ayuda a modularizar mejor cosas como el typing, el acceso a datos y la organización por librerías, haciendo que el proyecto escale de forma más limpia.

También me quedó claro que una de las mayores ventajas del monorepo es poder integrar no solo el frontend, sino también el backend en un mismo lugar. Me mostró cómo esto mejora la colaboración entre equipos, facilita los releases coordinados y permite que todo el código esté sincronizado. Incluso destacó que en tiempos de IA esto tiene todavía más valor, porque al tener frontend y backend en un solo repositorio, una herramienta de AI puede entender mucho mejor el contexto completo del proyecto. Además, me pareció interesante que NX ya no se limita a Angular: aunque nació muy ligado a ese ecosistema, ahora soporta muchas tecnologías como React, Express, Nest, Java y hasta .NET, lo que abre posibilidades para arquitecturas híbridas en un solo workspace.

Durante la demostración, vi el proceso práctico de crear un workspace vacío con NX, agregar el plugin de Angular y luego generar dos aplicaciones distintas: una para administración y otra para el website. Me gustó cómo explicó que cada app puede evolucionar con necesidades distintas —por ejemplo, una con SSR y otra no, o con distintas librerías visuales— sin estorbarse entre sí. Después mostró cómo crear librerías compartidas, como una de `data-access` para centralizar servicios que consumen APIs, y otra de `types` para compartir interfaces y modelos entre frontend, backend y distintas apps. Eso me ayudó a entender mejor la verdadera potencia del monorepo: no es solo guardar todo junto, sino estructurarlo de manera que el código compartido tenga límites claros y reutilizables.

Por último, aprendí que NX unifica la forma de trabajar todos esos proyectos con comandos consistentes para servir, compilar, testear o inspeccionar dependencias. También vi cómo agregar una API en NestJS al mismo repositorio, y cómo usar `nx graph` para visualizar gráficamente las relaciones entre apps y librerías. En conjunto, me llevé la idea de que NX es una herramienta muy poderosa para equipos y proyectos que ya están creciendo, especialmente cuando aparece duplicación de código, múltiples aplicaciones o la necesidad de centralizar frontend y backend. No lo percibí como algo obligatorio para cualquier proyecto pequeño, sino como una solución que empieza a brillar cuando la complejidad real del codebase lo justifica.
